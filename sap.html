SAP

(20 weeks)

Team

4 Software Engineers,

My Role

UI/UX Research & Design

Background

For our senior capstone project, we worked with a sponsor from SAP over the course of 5 months to create a free, open source transcription tool.

Objective

Create a web application that will allow interviewers to record and transcribe voice recordings, add labels for easy analysis, and provide privacy for certain phrases that are confidential.


Research

Competitive Analysis

To begin our research, we compiled a competitive analysis to compare existing transcription websites and see what features we can integrate into our web app. We looked extensively into their strengths, weaknesses, and pricing.

User Interviews

Once we completed the competitive analysis, we furthered our research by conducting user interviews with 4 PhD/graduate students. The goal of the interviews was to understand the user's perspective and what their struggles are when using current transcription sites. We aim to alleviate their pains of manually transcribing audio files and editing the audio to censor sensitive information.

We asked open-ended questions such as:
· What are some voice transcription tools you have heard of?
· What are some situations where you would use a voice transcription tool, what is your goal?
· What is your current challenge when transcrbing interviews (or any other audio recording)?
· What benefits do you see with the use of a voice transcription tool?
· What issues do you see in using a voice transcription tool?
· What features would you like it to improve upon?

Define

User Personas

Based on the information gathered from our research, we illustrated the goals, motivations, and frustrations of two ideal personas that would be utilizing our transcription tool.

Storyboard

To visualize possible situations where a user would need to use a transcription tool, we created two storyboards to play out two different scenarios.

1. User uploading audio file and being given transcription from audio file

2. User altering voice in audio recording

Product Features

With ideas and suggestions from our competitive analysis and user interviews, we curated a list of possible product features prioritized by what is needed, nice to have, and can be added later on.


Develop

Wireframes

Knowing which features we needed to implement, we created a rough sketch of what our web application will look like and the necessary steps a user will go through while using the transcription tool.

Goal: upload an audio file to a project

Goal: add a code and highlight certain information

Low Fidelity Mockups

Using the sketches above as a guide, we created low fidelity mockups of the product in more detail at each step.

Adding a project
Before a user can upload an audio file, they must create a project. This project will act as a folder to store all related files.

Uploading a file
When a user uploads a file, a progress bar will appear on the top left corner of the screen. This will let the user know how much longer it will take, and when it is complete. Once the upload is successful, the transcript will automatically show up beneath the toolbar. The different speakers will be identified by numbers with the timestamps at which they spoke at.

Adding codes, comments, and highlighting
Users can add codes similarly to how they add projects. Each code will have its own color. They can then use the highlight tool to select corresponding information. Additionally, users can add comments at specific time stamps and view them when they hover over the line of text. Each section of the transcript will also have a marker where users can check off what they have already gone through.

Minimizing projects menu
If the user wants to have more space to read through their transcript, they can minimize the side menu by pressing on the arrow at the top of the screen.

Usability Testing and Feedback

Once we had our low fidelity mockups completed, we conducted usability testing with the same 4 participants from our user interviews. Their task was to upload an audio file, add codes, and highlight key phrases from the transcript. Despite the positive feedback that we received, the users did have some suggestions on areas that we can improve on.

Positive Feedback
· Liked the labels
· Adding/uploading a project is obvious, clear and understandable

Improvements
· Add description page for what each file is
· Need clean screen for editing (move projects to separate page to reduce clutter)
· Snip tool should cut out portions and create new copy, don't want to alter actual raw data
· Ability to do multiple files at once
· Need error messages/help if something goes wrong
· Confused between code vs highlight
· Not sure what it means/what to do after creating the code label
· Not being able to actually type (but this is just a prototype so it's okay)
· Thought had to click on project 1 to upload file (but prototype automatically puts file into recently created project)
· Arrows: if nothing within it, don’t point down as if it is expanded, this was confusing

High Fidelity Mockups

Taking user feedback into consideration, we made the neccessary adjustments and improved our final design.

Projects Page
We moved the projects into their own separate page where users can add descriptions and color code each folder. They can now upload multiple files into specific projects, and once completed, will be listed underneath.

Transcript Page
For the transcript page, we moved the toolbar from the top of the page to the side for easier access. We also changed the appearance of the audio player and added a 10 second forward/backward option. Rather than having the user create a code, and then selecting the hightlight option, we just have the user select the text they want to highlight and add a code as one single step.

Key Takeaways

This project was an exciting and great learning experience for me since it was my first time working on UI/UX for a product through its entire lifecycle.

Communication is key
About halfway into the project, the covid-19 lockdown occurred. Because of this, everything was converted to a virtual workspace. Constant communication between the team was very important because we went from meeting and working together in person every week, to strictly online only which caused some miscommunication and confusion amongst the team and each person's tasks. To ensure that each team member knew what they were doing and planning to do in the future, we scheduled daily check-ins on slack and weekly zoom calls.

Validation of features
Clarifying the product details and specifying its requirements early in the design process was important for us to be efficient as possible and make sure that the product was correct. We had frequent check-ins with the sponsor about the features we were implementing to confirm that they were still desired.

Appreciating criticism
Asking for feedback from user testing, my peers, and my sponsor was crucial for me to push my creativity and to design the UI as best I could for potential users. Their critiques helped me see things that I was missing or had second-guessed myself on. I realized that no design is perfect and that there is always room for improvement.

If we had more time...
As you can see from the video above, our final product looked a bit different from our final mockup design. Because of the software we were using, we experienced trouble adding in some features, such as the toolbar, which we made as a right-click menu instead. Due to the time constraint, we didn't have enough time to clean up the UI as much as we would have liked. The labels on the side for each highlighted text weren't aligned on the side and we weren't able to add different color options for them. However, we did the best we could as making sure that the transcript/audio functions actually worked was the biggest priority.
